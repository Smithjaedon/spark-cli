Stack Phases Concepts

Build better projects with spark

A Python CLI scaffolding tool that generates production-ready project structures. Type-safe, progress bars, templates for API and basic projects — all built with code quality as the foundation.

Python 3.12+ ~1 hour phases 4 phases uv tool install

Tech stack

Everything free-tier. No paid services, no credit cards required.

Typer → CLI framework

Industry-standard Python CLI tool. Type-hint driven, built on Click. Handles arguments, subcommands, help text — everything spark needs.

Rich → Terminal UI

Progress bars, colored prompts, styled panels. Gives spark a professional feel without a heavy TUI framework.

uv → Project init & deps

Spark delegates to uv init and uv add for pyproject generation. No shipping stale config files — always up to date.

pathlib → Filesystem

Python's built-in path library. Handles recursive directory creation, file writing, path resolution. No dependencies needed.

pytest → Testing

Typer's CliRunner for CLI tests, tmp_path fixtures for filesystem isolation. Fast, reliable, zero config.

ruff → Linting

Single-tool linting + formatting. Configured in pyproject.toml. Runs in CI. Catches everything before it ships.

Commands

v1 ships two commands. Everything else is future scope.

spark create <name>
Interactive scaffolding wizard. Pick a template type (API, basic), then spark generates the full project tree with a progress bar. Creates a new directory for the project.
spark dev
Wrapper around process-compose up. Detects process-compose.yaml in the project root and runs it. Saves remembering the exact command.
spark add <thing>
Future command. Inject add-ons into an existing scaffolded project. Not designed yet — explicitly deferred to post-v1.

Phases

Four sequential phases, each ~1 hour. Each ends with something working.

1

Project skeleton + scaffolding engine

Planned~1 hour

Get spark create working end-to-end. The scaffolding engine is the foundation — if it's clean, every feature after is just adding templates or commands. Prototyper V2's problem was mixing concerns; this phase enforces separation from day one.

What to build

  • Initialize project: uv init spark, add typer + rich deps
  • Create module structure: cli.py, app.py, scaffold/, ui/, utils/
  • Implement Template dataclass + two template definitions (api, basic)
  • Implement scaffold engine: recursive dir creation, empty __init__.py files, stub files, uv init + uv add subprocess
  • Implement Rich-based interactive promoter (template type selection)
  • Implement Rich progress bar during scaffolding
  • Wire up cli.py → app.py orchestration
  • Set up Rich logging handler with SPARK_LOG_LEVEL env var
  • Error handling: ProjectExistsError, TemplateNotFound, styled error output

Key decisions

Stub files are plain Python strings (no Jinja in v1). Empty __init__.py is default for all Python directories. uv init/add runs as subprocess — errors bubble up as ScaffoldError. Functions under 20 lines, one job each.

Quality bar

Every module exports a clean public API via __init__.py. Type hints on every function. Zero print() — all output through Rich Console. Tests pass with tmp_path isolation.

2

Template stubs + spark dev

Planned~1 hour

Templates need real content so scaffolding produces a runnable project, not empty folders. spark dev closes the loop — create, then immediately run.

What to build

  • Write stub content for all API template files (main.py, core/*, models.py, schemas.py, services/*, ws/*)
  • Write stub for Basic template (src/main.py as Typer CLI entry point)
  • Write process-compose.yaml stub for API template (uvicorn + redis)
  • Implement spark dev — detects process-compose.yaml, runs process-compose up
  • Write .env.example and .gitignore stubs

Key decisions

Stub files contain real working code — NOT placeholder comments. An API main.py actually starts uvicorn. Basic main.py runs a typer app. This makes spark create → spark dev a valid workflow.

Quality bar

Every stub file is valid Python (no syntax errors). Import chains work across modules. Basic template has zero references to FastAPI, auth, database, or websockets.

3

Testing, CI, polish

Planned~1 hour

This is where "code quality" stops being aspirational and becomes enforced. Every change from here on has a guardrail — tests, linting, CI.

What to build

  • CLI integration tests: create project, assert tree matches expected, existing dir error, dev with/without compose file
  • Unit tests: template definition validation, scaffold engine with mocked filesystem
  • pytest config in pyproject.toml + conftest.py fixtures
  • Ruff configuration in pyproject.toml
  • GitHub Actions CI: matrix (3.12, 3.13), ruff check, pytest, uv build
  • UX polish: color-coded output, success panel, suggested next steps, version flag, clean --help

Key decisions

Tests use Typer's CliRunner (no subprocess). Mock uv init/add calls in CI. Ruff config in pyproject.toml — single source of truth, no separate config files.

Quality bar

ruff check passes with zero errors. All tests pass. Coverage >80% for scaffold engine. CI is green on every push.

4

Distribution + documentation

Planned~1 hour

Distribution is the last mile. If it's not easy to install and understand, the tool doesn't exist. This phase polishes the edges so anyone (or future-you) can install and use spark immediately.

What to build

  • Configure pyproject.toml entry point: spark = "spark.cli:app"
  • Write README.md: what, quick start, templates, commands reference
  • Test uv tool install . from repo root
  • Test spark create from clean environment (outside repo)
  • Write CONTRIBUTING.md — template authoring guide, dev setup

Key decisions

uv tool install is the only distribution channel — no homebrew, no npm, no PyPI needed (though optionally supported). README is the primary documentation — keeps maintenance surface small.

Quality bar

uv tool install spark succeeds from any machine. spark --help shows clean command list. README can be read in 2 minutes and gets you running in 30 seconds.

Template specifications

Two templates in v1. The directory tree each produces.

API P0
Full FastAPI project with app/, core/ (auth, database, exceptions, logging), routers/, services/, ws/, tests/{unit,integration}, scripts/, process-compose.yaml. Dependencies: fastapi, sqlalchemy, asyncpg, pyjwt, redis, pwdlib.
Basic P0
Minimal Python project with src/ layout. Single entry point (Typer CLI), tests/{unit}, scripts/. No app/, no core/, no database/auth/ws layers. Dependencies: typer, rich, pytest.

Teaching concepts

Senior-level patterns woven into each phase. Learn by building.

Separation of concerns

Templates own what to build. Engine owns how. CLI owns when. UI owns the look. Change one without touching the others — Prototyper V2's core sin was mixing these.

Phase 1

Minimal public API

Each module's __init__.py exports only what consumers need. Hidden internals stay private. Keeps the surface small and refactoring safe.

Phase 1

Exception hierarchy, not exception soup

One base SparkError. Subclasses for each failure mode. Top-level handler catches them all. Never catches bare Exception. Never scatters error handling throughout the code.

Phase 1

Dependency injection

Pass Console and Path to the scaffold engine instead of importing them globally. Makes the engine testable without a TTY and replaceable without side effects.

Phase 1

Composition over inheritance

Templates are dataclasses, not subclasses. A template is a value with structure, not a class hierarchy. Adding a template means adding a value, not extending a base class.

Phase 2

Testing pyramid

Unit tests for engine logic (fast, many). Integration tests for CLI invocation (medium, critical paths). Manual testing for visual UX (rare). Know which layer each test belongs to.

Phase 3

Config as code

Ruff config lives in pyproject.toml, not a separate file. Single source of truth, one place to look, one file to edit — reduces config drift.

Phase 3

Distribution as UX

uv tool install spark is the entire install story. No brew, no npm, no manual PATH management, no pipx dance. The install experience is the user experience — make it invisible.
